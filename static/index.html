Windows System Memory Breakdown

Windows does not allocate memory in a simple "Used" or "Free" binary structure. High RAM usage often stems from active caching or driver allocations.

In-Use Memory

0.00 GB

RAM allocated actively to user applications and Windows processes. This memory is completely committed and cannot be easily reclaimed without closing apps.

Standby List (System Cache)

0.00 GB

Memory used to store files and programs you previously opened. Windows keeps them in RAM to speed up access. **This memory is immediately released** if a running app demands it, which is why it shouldn't be counted as a leak.

Non-Paged Pool

0.00 GB

Reserved kernel/driver memory that *must* remain in RAM and cannot be moved to the disk pagefile. If this exceeds **1.5 GB**, you likely have a **kernel memory leak** caused by a buggy driver (often Wi-Fi, Ethernet, or graphics).

Paged Pool

0.00 GB

Kernel memory used by system services that can be paged out to the disk if RAM gets tight. Managed automatically by the operating system.

WSL2 & Container Virtualization Analyzer

Windows Subsystem for Linux (WSL2) runs within a utility Virtual Machine (represented by the vmmemWSL or vmmem process). It dynamically allocates RAM but does *not* release it back to Windows automatically when tasks finish.

Docker Desktop is Locking WSL

Docker Desktop is currently active in the background. It maintains an active connection to WSL and will immediately wake the virtual machine back up if it is terminated. To successfully reclaim this memory, please Quit Docker Desktop (right-click the Docker whale icon in your taskbar system tray and select Quit Docker Desktop), then click the shutdown button again.

WSL Configuration & Capping

To permanently limit WSL2 memory consumption (capping it so it never starves Windows), create a configuration file named .wslconfig in your user home folder (C:\Users\cbige\.wslconfig) with the following content: 

[wsl2]
memory=4GB
